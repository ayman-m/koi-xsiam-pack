#### Playbooks

##### Koi Unified - Execute Endpoint Script

Fixed a bug where every run silently targeted only 30 endpoints. The endpoint query did not pass a
`limit`, so `core-get-endpoints` applied its default page size of 30 — regardless of how many
endpoints were in the target group. On a 1000-endpoint group each run reached a rotating 30, and the
remainder were never scanned.
- Added a **max_endpoints** input (default 500) which is now passed as the query's `limit`. It can be
  overridden per List entry with `target.max_endpoints`.

Fixed false-positive failure notifications. `core-script-run` dispatches an action and then polls it;
when polling expired, the playbook treated that as an execution failure even though the action was
still running in Action Center and usually completed on its own.
- Added a **dispatch check** on the error path. If an `action_id` exists, the run is now reported as
  `timed_out` rather than failed.
- Added `ScriptResult.timed_out` and `ScriptResult.endpoint_count` outputs.

##### Koi Unified - Process Config Entry

- A timed-out run is now recorded as an informational **STILL RUNNING** entry instead of sending a
  failure email. Only a genuine dispatch failure — where no action was created — notifies.
- Notification emails now include the endpoint count and the Action ID. The previous failure mail gave
  only a reason string, with no way to find the corresponding action.
- The failure subject is now **FAILED TO DISPATCH**, to distinguish it from a run that started and is
  still in progress.
- Passes `target.max_endpoints` through to the execution sub-playbook.
